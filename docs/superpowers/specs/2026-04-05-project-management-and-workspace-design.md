# Project Management And Workspace Design

## 목적

이 문서는 대화형 광고 시나리오 AI 멀티에이전트 시스템의 초기 제품 경험을 다시 정의한다. 범위는 다음 두 화면이다.

- 프로젝트 관리 탭
- 프로젝트 내부 워크스페이스

이 설계의 목표는 사용자가 프로젝트를 빠르게 열고, 채팅 중심으로 작업을 지시하면서도, 좌측 상태 레일을 통해 현재 시스템 진행 상황과 병목을 이해할 수 있게 만드는 것이다.

## 배경과 재설계 이유

기존 설계는 아래 이유로 제품 논리와 맞지 않았다.

- 좌측에 일부 에이전트만 전면 노출되어 전체 시스템 구조를 잘못 추상화했다.
- 워크스페이스가 사용자의 지시 공간인지, AI 자동 진행 대시보드인지 모호했다.
- 병렬 에이전트 시스템의 복잡성이 사용자에게 직접 드러나면서도, 실제 의사결정 흐름과는 연결되지 않았다.

따라서 이번 재설계는 `개별 에이전트 전면 노출`을 버리고, `채팅 중심 + 작업 상태 레일 + 상세 팝업` 구조로 정리한다.

## 핵심 원칙

- 인터랙션 모델은 `chat-first`다.
- 사용자는 기본적으로 계속 지시권을 가진다.
- 시스템은 다음 행동을 제안할 수 있지만, 사용자의 명시적 지시나 승인 없이 큰 단계를 알아서 넘기는 느낌을 주면 안 된다.
- 기본 화면은 `개별 에이전트 목록`이 아니라 `작업 상태`를 보여준다.
- 내부 에이전트 정보는 기본 숨김이며, 필요할 때만 팝업에서 드릴다운한다.
- 병목, 사용자 선택 대기, 진행 중 상태는 한눈에 보여야 한다.

## 정보 구조

제품의 1차 정보 구조는 2단계다.

1. `프로젝트 관리 탭`
2. `프로젝트 내부 워크스페이스`

사용자 흐름은 아래와 같다.

1. 사용자가 프로젝트 관리 탭에서 프로젝트를 생성하거나 기존 프로젝트를 연다.
2. 프로젝트 카드를 클릭하면 워크스페이스로 진입한다.
3. 워크스페이스 첫 화면은 환영문이 아니라 `현재 상태 요약 + 다음 가능한 행동`으로 시작한다.
4. 사용자는 채팅으로 지시하고, 시스템은 상태와 결과를 피드에 구조적으로 반영한다.

## 화면 1: 프로젝트 관리 탭

### 역할

프로젝트 관리 탭의 역할은 두 가지뿐이다.

- 새 프로젝트를 만든다.
- 기존 프로젝트에 다시 들어간다.

이 화면은 운영 대시보드가 아니라 `프로젝트 런처`다.

### 레이아웃

- 상단: 브랜드/앱 헤더
- 우상단: `Create Project` 버튼
- 본문: 폴더형 프로젝트 카드 그리드

프로젝트 생성은 최소 입력만 사용한다. 첫 구현에서는 `프로젝트 이름`만 받는다.

### 프로젝트 카드

프로젝트 카드에는 아래 정보만 노출한다.

- 최근 산출물 대표 썸네일
- 프로젝트 이름
- 마지막 활동 시각
- 현재 작업 상태 한 줄
- 사용자 액션 필요 여부가 드러나는 상태 문구

상태 문구 예시:

- `Drafts ready`
- `Waiting for concept decision`
- `Scenario blocked by concept`
- `Storyboard in progress`

썸네일 규칙은 아래와 같다.

- 최근 산출물이 이미지/콘티면 그 결과를 대표 이미지로 사용한다.
- 아직 시각 결과가 없고 드래프트/컨셉 단계면 해당 결과를 대표하는 커버 스타일 썸네일을 사용한다.
- 산출물이 전혀 없으면 초기 플레이스홀더 썸네일을 사용한다.

카드 정렬은 `최근 작업순`을 기본으로 하되, `Waiting for user` 성격의 프로젝트를 상단에서 더 강하게 강조한다.

여기서는 개별 에이전트 정보를 노출하지 않는다.

### Empty State

프로젝트가 없을 때는 “첫 프로젝트를 만들어 시작”하는 empty state를 보여준다. 설명은 짧게, 행동 유도는 강하게 가져간다.

## 화면 2: 프로젝트 내부 워크스페이스

### 역할

워크스페이스는 `채팅 중심 작업 공간`이다. 사용자는 여기서 계속 자연어로 지시하고, 시스템은 결과와 다음 행동을 제안한다.

### 기본 레이아웃

기본 구조는 3영역이다.

- 좌측: 작업 상태 레일
- 중앙: 채팅 피드
- 우측: 현재 포커스 결과물 패널

중앙 채팅 피드가 항상 시각적 주인공이어야 한다.

## 좌측: 작업 상태 레일

### 기본 개념

좌측은 `에이전트 목록`이 아니라 `작업 상태 레일`이다. 사용자는 이 패널을 통해 지금 시스템이 어떤 종류의 일을 하고 있고, 어디서 막혔는지 빠르게 이해한다.

### 작업군 단위

기본 작업군은 아래처럼 잡는다.

- `Research`
- `Drafts`
- `Concept`
- `Scenario`
- `Storyboard`
- `QA`

이 작업군은 사용자가 이해하는 단위이며, 내부적으로는 여러 에이전트가 이 안에서 병렬로 동작할 수 있다.

### 상태 체계

각 작업군은 아래 상태만 사용한다.

- `Running`
- `Waiting for user`
- `Blocked`
- `Ready`
- `Done`

예시:

- `Research — Running`
- `Drafts — Ready`
- `Concept — Waiting for user`
- `Scenario — Blocked`

### 상태 의미

- `Running`: 시스템이 현재 처리 중
- `Waiting for user`: 사용자의 선택, 승인, 추가 지시 필요
- `Blocked`: 앞선 단계 결정이 없어 아직 시작할 수 없음
- `Ready`: 검토 가능한 결과물이 준비됨
- `Done`: 현재 라운드 기준 완료

### 상세 팝업

사용자가 작업군을 클릭하면 `작업 상세 팝업`이 열린다.

팝업에는 아래 정보만 포함한다.

- 작업군 이름
- 현재 상태
- 왜 이 상태인지
- 내부적으로 사용된 에이전트 목록
- 최근 산출물
- 다음 필요 액션

예:

- `Drafts`
  - 현재 상태: `Ready`
  - 이유: `3개의 드래프트가 생성되어 비교 가능`
  - 사용된 에이전트: `Copy Draft Workers`, `Copy Curator`, `Memory Agent`
  - 최근 산출물: `Draft A`, `Draft B`, `Draft C`
  - 다음 필요 액션: `하나를 선택하거나 조합 방향을 지시`

이 팝업은 디버그 콘솔이 아니라 `작업 인스펙터`여야 한다.

## 중앙: 채팅 피드

### 역할

채팅 피드는 단순한 메시지 로그가 아니라 `의사결정 기록`이다. 사용자의 지시와 시스템의 해석, 상태 변화, 결과물이 시간 순으로 기록된다.

### 버블 규칙

- 사용자 메시지: 오른쪽
- 시스템 메시지: 왼쪽

### 피드 오브젝트 종류

피드 안의 오브젝트는 최소 4종류다.

- 사용자 메시지
- 시스템 메시지
- 상태 블록
- 결과물 카드

각 타입은 시각적으로 명확히 구분되어야 한다.

### 기본 흐름 패턴

피드의 기본 순서는 아래와 같다.

1. 사용자 지시
2. 시스템의 해석 또는 다음 행동 제안
3. 중간 상태 메시지
4. 결과물 카드
5. 다음 가능한 행동 제안

예:

1. `러닝 브랜드 광고 방향을 탐색해줘`
2. `먼저 리서치와 드래프트 생성을 진행할 수 있습니다.`
3. `Research running`
4. `리서치 요약 카드`, `드래프트 3안 카드`
5. `하나를 선택하거나 두 안을 조합하도록 지시할 수 있습니다.`

### 첫 진입 시 시작 상태

프로젝트 진입 직후 첫 시스템 메시지는 환영문이 아니라 아래 구조를 따라야 한다.

- 현재 상태 요약
- 최근 결과 요약
- 다음 가능한 행동

예:

`드래프트 3안이 준비되어 있습니다. 하나를 선택하거나, 두 안을 조합하도록 지시할 수 있습니다.`

### 병목과 대기 노출

`Waiting for user` 또는 `Blocked` 상태는 좌측 레일뿐 아니라 피드 안에서도 한 번 더 드러나야 한다.

예:

`현재 컨셉 승인을 기다리고 있습니다. 승인하거나 수정 방향을 지시하세요.`

## 우측: 현재 포커스 결과물 패널

우측 패널은 상시 대시보드가 아니다. `지금 대화의 대상이 되는 결과물`만 붙잡는 보조 영역이다.

예:

- 선택된 드래프트
- 현재 승인 대기 중인 컨셉
- 시나리오 컷 구성
- 콘티 프리뷰

결과물이 없으면 비어 있거나 접힌 상태로 둔다.

## 게이트와 승인 UX

### 원칙

게이트는 기존처럼 유지하되, UX적으로는 `자동 파이프라인 승인`보다 `AI가 제안한 다음 행동에 대한 명시적 허가`에 더 가깝게 느껴져야 한다.

### 표시 방식

게이트가 필요한 순간에는 채팅 피드 하단에 `Approve Bar`가 등장한다.

포함 요소:

- 현재 승인 대상 요약
- `수정 요청`
- `승인 →`

예:

- `드래프트 3안 정리 완료`
- `컨셉 확정안 준비됨`
- `5컷 시나리오 검토 필요`
- `최종 콘티 검토 필요`

### UX 규칙

- Approve Bar는 항상 떠 있는 것이 아니라, 실제 결정이 필요한 순간에만 나타난다.
- 채팅을 대체하지 않는다.
- 사용자가 버튼 대신 채팅으로 지시해도 동일한 의미로 처리할 수 있어야 한다.

## 상태 전환 규칙

1. 사용자가 요청을 보낸다.
2. 시스템은 해석 후 가능한 다음 행동을 제안하거나 바로 실행을 선언한다.
3. 실행 중이면 좌측 레일과 피드 모두에 상태를 반영한다.
4. 결과가 준비되면 결과물 카드와 `다음 가능한 행동`을 함께 보여준다.
5. 사용자 결정이 필요하면 작업군 상태는 `Waiting for user`가 된다.
6. 앞선 결정이 없어 후속 작업을 시작할 수 없으면 `Blocked`가 된다.

## 구현 시 주의사항

- 좌측 패널을 개별 에이전트 목록으로 다시 퇴행시키지 않는다.
- 작업 상태 레일은 항상 사용자 행동 판단에 도움이 되는 문구만 보여준다.
- 상세 팝업은 설명 가능성을 높이되 디버그 패널처럼 보여서는 안 된다.
- 채팅 피드 안에서 상태 메시지와 결과물 카드는 시각적으로 엄격히 구분되어야 한다.
- 프로젝트 관리 탭은 끝까지 가볍게 유지한다.

## 결정 요약

- 루트는 `프로젝트 관리 탭`이다.
- 새 프로젝트는 최소 입력으로 생성한다.
- 프로젝트 카드는 `현재 작업 상태 한 줄`만 보여준다.
- 워크스페이스는 `좌측 작업 상태 레일 / 중앙 채팅 / 우측 포커스 결과물` 구조다.
- 좌측은 에이전트 목록이 아니라 `작업군 상태`를 보여준다.
- 작업군 클릭 시 `상세 팝업`이 열린다.
- 채팅 피드는 `지시 → 해석/제안 → 상태 → 결과 → 다음 행동` 순서를 따른다.
- 병목과 사용자 선택 대기 상태는 좌측과 피드 양쪽에서 드러난다.
- 게이트는 필요할 때만 `Approve Bar`로 나타난다.
